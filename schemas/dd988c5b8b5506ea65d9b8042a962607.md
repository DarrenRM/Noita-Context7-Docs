# dd988c5b8b5506ea65d9b8042a962607

This XML file defines the schema for various game components in Noita. It acts as a blueprint, specifying the structure and data types for different aspects of game entities, such as AI behavior, physics, item properties, and more. These schemas are crucial for the game engine to correctly interpret and utilize the data associated with each game object.

## File Structure

The file is structured as a collection of `<Component>` elements, each representing a distinct game component. The root element is `<Schema>` which contains a `hash` attribute.

Each `<Component>` element has a `component_name` attribute that identifies the type of component it defines. Inside each `<Component>`, there are multiple `<Var>` elements.

Each `<Var>` element represents a variable or property within that component and has the following attributes:
*   `name`: The name of the variable.
*   `size`: The size of the variable in bytes.
*   `type`: The data type of the variable (e.g., `int`, `float`, `bool`, `class std::basic_string<char,struct std::char_traits<char>,class std::allocator<char> >`, `struct LensValue<float>`, `__int64`, `unsigned int`).

The `<Var>` elements themselves are typically empty, indicating that their values are defined elsewhere in the game's data files.

## Key Patterns

*   **Component-Based Architecture:** The file clearly demonstrates a component-based design, where different functionalities are encapsulated into distinct components (e.g., `AIAttackComponent`, `AIComponent`, `PhysicsBodyComponent`).
*   **Variable Definitions:** Each component is defined by a list of its constituent variables, specifying their names, sizes, and data types. This allows for a structured and predictable way to store and access component data.
*   **Data Type Variety:** A range of data types are used, from fundamental types like `int`, `float`, and `bool` to more complex types like `class std::basic_string` (likely for string data) and custom structs like `struct LensValue<float>`.
*   **Naming Conventions:** Variable names often reflect their purpose, such as `use_probability`, `min_distance`, `gravity`, `cost`, `stealable`, `speed_threshold`, and `damage_multiplier`.
*   **Boolean Flags:** Many variables are of type `bool`, serving as flags to enable or disable specific behaviors or features (e.g., `attack_ranged_enabled`, `flying_needs_recharge`, `stealable`, `is_enabled`).
