# ae86cb47149cc8b51503f1d1916be779

This file defines the data structures and properties for various game components in Noita. It acts as a schema, outlining the types and names of variables that make up different game entities, such as AI behaviors, item properties, and material interactions. This schema is crucial for the game engine to correctly load, interpret, and utilize the data associated with these components, enabling the creation of diverse enemies, items, and environmental effects.

## File Structure

The file is structured as an XML document. The root element is `<Schema>`, which contains multiple `<Component>` elements. Each `<Component>` element represents a distinct game component and has a `component_name` attribute that identifies it.

Inside each `<Component>`, there are `<Var>` elements. Each `<Var>` element defines a single variable within that component. These `<Var>` elements have the following attributes:

*   **`name`**: The name of the variable.
*   **`size`**: The size of the variable in bytes.
*   **`type`**: The data type of the variable (e.g., `int`, `float`, `bool`, `class std::basic_string<char,struct std::char_traits<char>,class std::allocator<char> >`).

The `<Var>` elements themselves are empty, indicating that this file defines the *structure* of the data, not its specific values.

## Key Patterns

*   **Component-Based Design**: The data is organized around distinct game components, suggesting a modular approach to game entity design.
*   **Variable Definitions**: Each component is broken down into individual variables with explicit type and size information.
*   **String Representation**: String types are represented by a complex class name, `class std::basic_string<char,struct std::char_traits<char>,class std::allocator<char> >`, indicating they are standard C++ strings.
*   **Common Data Types**: `int`, `float`, and `bool` are frequently used for numerical and boolean properties.
*   **AI Components**: Several components appear to be related to AI behavior, such as `AIAttackComponent` and `AIComponent`, defining parameters for movement, combat, and decision-making.
*   **Item/Inventory Properties**: Components like those related to `item_recoil` and inventory management (`auto_pickup`, `uses_remaining`) suggest definitions for how items function and are handled.
*   **Material Interactions**: Components like `MaterialSuckerComponent` indicate definitions for how game elements interact with different materials.
